The Documentation Repository

Most source for published Z-Mail (and related) product documentation, for all platforms including Windows, is presently stored on zyrcon.diego.netmanage.com in the directory /usr1/build/DocRep. This directory is colloquially referred to as the DocRep.

Within the DocRep are directories named for documentation projects. Of these, only a few are of direct interest to the Z-Mail developer:

  • nls - Network License Server
  • mac - ZM-Mac
  • sgihelp - MediaMail (help books)
  • windows - ZM-Win
  • zmail - Z-Mail (UNIX/Motif)
  • zmlite - Z-Mail Lite
  • zpop - ZPOP server

The doc.sh script looks for documentation in only three of these:

  • nls - version specified in doc.sh
  • zmail - version extracted from shell/version.c
  • zmlite - version extracted from shell/version.c

Each of these three directories must contain subdirectories whose names match the version-string values used by doc.sh when building a release distribution of any Z-Mail product for UNIX.

Within the version subdirectories, doc.sh looks for the following directories and files:

  • zmail and zmlite
    • Copyrights/Copyrights.txt
    • ig/Install.ps
    • ig/Install.txt
    • License/License.ps
    • License/License.txt
    • README
    • Relnotes/Relnotes.ps
    • Relnotes/Relnotes.txt
  • zmail only (Motif)
    • zmail.1
      This probably should be included with Lite as well.
  • zmlite only
    • README.doc
    • RefCard/card.ps
    • TermConfig/TermConfig.ps
    • TermConfig/TermConfig.txt
    • ug/Guide.ps
  • nls only (currently included with both Motif and Lite releases)
    • ig/NLSinstall.ps
    • ig/NLSinstall.txt

An error is generated if any of these files is not found. All of these files are copied out of the DocRep by doc.sh, regardless of what product distribution of Z-Mail is being built. However, the version-string need not be the same for Motif and Lite, so if the product releases are not in sync either of these may refer to an older existing version of the documentation.

If documentation for a given release is partially completed, e.g. the release notes are finished for a beta release but other documentation has not been updated, then the version subdirectory should be created to hold the completed documents and then populated with symbolic links to the most recent older version directory to make the other required documents available.
